# Auto-Inventory — File Tracking Protocol

> Runs at: session end, after significant file creation, on /mxm-release
> Writes to: .claude-sessions-memory/file_inventory.md

---

## Purpose

Automatically tracks all architecturally significant files in the project. Prevents inventory drift across sessions by generating a diff-based update at session end.

---

## Trigger Conditions

Run inventory update when ANY of these occur:
- Session end protocol begins (Step 1 — before any other memory writes)
- User runs `/mxm-release` (inventory snapshot before version bump)
- 5+ new files created in a single session
- User says "update inventory", "what files do we have", "file audit"

---

## Inventory Generation Protocol

### Step 1 — Scan for significant files

Scan the project root (excluding `node_modules/`, `.git/`, `community-packs/`, `__pycache__/`). A file is significant if it matches ANY of these:

**Always track:**
- `config/*.json` — project manifests, registries
- `documents/architecture/*` — PRD, FRD, SRD, ARCHITECTURE.md
- `documents/business/*` — investor narrative, financial models
- `*.md` in project root — CLAUDE.md, CLAUDE.project.md, README.md, CHANGELOG.md
- `.claude/commands/*.md` — slash commands
- `.claude/skills/*/SKILL.md` — skill definitions
- `agents/**/*.md` — agent definitions (if present)
- `docker-compose.yml`, `Dockerfile` — infrastructure
- `package.json`, `turbo.json`, `pyproject.toml` — build config
- `*.yml` in `.github/workflows/` — CI/CD
- `openapi.yaml`, `schema.prisma`, `*.sql` — API/DB schemas
- `tsconfig.json`, `tailwind.config.*` — tooling config
- `.env.example`, `.env.template` — env templates (NOT .env itself)

**Track if new this session:**
- Any file the planner, implementer, or reviewer created
- Any file explicitly mentioned in task_plan.md

**Never track:**
- `node_modules/`, `.git/`, `__pycache__/`, `.next/`, `dist/`, `build/`
- `.env`, `.env.local` (secrets)
- `documents/architecture/.secrets/*` (secrets)
- Binary files (images, fonts, compiled assets)

### Step 2 — Diff against existing inventory

Read `.claude-sessions-memory/file_inventory.md` if it exists.
- Files in scan but NOT in inventory → mark as `ADDED`
- Files in inventory but NOT in scan → mark as `REMOVED`
- Files in both with different metadata → mark as `MODIFIED`
- Files in both unchanged → keep as-is

### Step 3 — Write updated inventory

Write `.claude-sessions-memory/file_inventory.md`:

```markdown
# File Inventory — [project.name]

> Auto-generated by Maxim session-memory auto-inventory
> Last updated: [YYYY-MM-DD]
> Total tracked files: [count]

## Project Configuration

| File | Path | Purpose | Last Session |
|---|---|---|---|
| project-manifest.json | config/ | Project identity, compliance, tech stack | [date] |
| agent-registry.json | config/ | Agent catalog | [date] |

## Architecture Documents (documents/architecture/)

| File | Path | Purpose | Last Session |
|---|---|---|---|
| ARCHITECTURE.md | documents/architecture/ | System architecture | [date] |
| PRD.md | documents/architecture/ | Product requirements | [date] |
| FRD.md | documents/architecture/ | Functional requirements | [date] |
| SRD.md | documents/architecture/ | System requirements | [date] |

## Business Documents (documents/business/)

| File | Path | Purpose | Last Session |
|---|---|---|---|
| [files found] | documents/business/ | [purpose] | [date] |

## Application Source

| File | Path | Purpose | Last Session |
|---|---|---|---|
| [entry points, key modules] | [path] | [purpose] | [date] |

## Infrastructure & CI/CD

| File | Path | Purpose | Last Session |
|---|---|---|---|
| [docker, workflows, IaC] | [path] | [purpose] | [date] |

## Changes This Session

| Action | File | Agent |
|---|---|---|
| ADDED | [path] | [planner/implementer/reviewer] |
| MODIFIED | [path] | [agent] |
| REMOVED | [path] | [reason] |
```

### Step 4 — Update reference_key_files.md

After writing file_inventory.md, sync any newly tracked files into `reference_key_files.md` under the appropriate category. The inventory is the scan source; reference_key_files is the curated lookup.

---

## Integration with Other Protocols

| Protocol | How Inventory Integrates |
|---|---|
| Session End | Inventory runs as Step 0 (before other memory writes) |
| `/mxm-release` | Inventory snapshot taken before version bump |
| `/mxm-plan` | Planner reads inventory to understand project structure |
| `/mxm-implement` | Implementer updates inventory after creating files |
| `/mxm-review` | Reviewer checks inventory for undocumented files |
| Auto-compact | Inventory survives compaction (it's in session-memory) |

---

## Version Drift Detection

While scanning, if the inventory detects version strings in files, it flags mismatches:

```
VERSION CHECK
  Source of truth: config/agent-registry.json → [version]
  README.md badge: [version] — [MATCH/MISMATCH]
  documents/guides/HELP.md header:  [version] — [MATCH/MISMATCH]
  CHANGELOG.md:    [version] — [MATCH/MISMATCH]
  package.json:    [version] — [MATCH/MISMATCH]
```

If ANY mismatch: warn the user and suggest running `bootstrap/sync-version.ps1` (or `.sh`).
